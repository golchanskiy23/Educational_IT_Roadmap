# Ручное управление памятью с помощью `unsafe` в Go

## Что такое пакет `unsafe`?

Пакет `unsafe` предоставляет операции, которые выходят за рамки безопасности системы типов Go. Он позволяет программам выполнять действия, которые обычно запрещены — например, напрямую манипулировать указателями памяти.

---

## Зачем использовать `unsafe`?

Несмотря на то что сборщик мусора и система типов Go обеспечивают безопасность разработки, бывают ситуации, когда ручное управление памятью необходимо:

1. **Производительность** — обход сборщика мусора в критически важных приложениях может дать значительный прирост скорости.
2. **Совместимость** — при работе с C-библиотеками может потребоваться прямая манипуляция памятью.
3. **Контроль расположения данных в памяти** — в некоторых сценариях (сериализация, сетевое взаимодействие) важно точно контролировать расположение данных в памяти.

---

## Выделение памяти через `unsafe`

С помощью cgo можно вызывать функции `malloc` и `free` из C для ручного управления памятью:
````````go
package main

import "C"

func main() {
    size := 10 // размер блока памяти в байтах
    ptr := C.malloc(C.size_t(size))
    defer C.free(ptr)

    // Использование выделенной памяти...
}
` ``

---

## Приведение типов через `unsafe`

`unsafe.Pointer` позволяет приводить указатели к различным типам. Это полезно, когда известно внутреннее представление типа в памяти:
```````go
var x int = 10
y := *(*float64)(unsafe.Pointer(&x))
// y получит неопределённое значение — память int интерпретируется как float64
` ``

---

## Внутреннее представление слайсов

Слайс в Go внутренне представлен структурой:
``````go
type slice struct {
    array unsafe.Pointer
    len   int
    cap   int
}
` ``

Зная это, можно манипулировать заголовками слайсов, например выполнять re-slicing без дополнительных аллокаций.

## Продвинутые примеры

### 1. Арифметика указателей

Go не поддерживает прямую арифметику указателей, как C/C++. Но через `unsafe` это возможно с конвертацией в `uintptr`:
`````go
arr := [5]int{10, 20, 30, 40, 50}
ptr := unsafe.Pointer(&arr[0])
nextPtr := unsafe.Pointer(uintptr(ptr) + unsafe.Sizeof(arr[0]))
nextValue := *(*int)(nextPtr)
// nextValue == 20
` ``

### 2. Доступ к полям структуры без reflect
````go
type MyStruct struct {
    A int
    B float64
}

s := MyStruct{A: 5, B: 3.14}
bPtr := (*float64)(unsafe.Pointer(uintptr(unsafe.Pointer(&s)) + unsafe.Offsetof(s.B)))
// *bPtr == 3.14
` ``

### 3. Union-подобное поведение
```go
type Union struct {
    data [8]byte
}

func (u *Union) SetInt64(val int64)     { *(*int64)(unsafe.Pointer(&u.data[0])) = val }
func (u *Union) GetInt64() int64        { return *(*int64)(unsafe.Pointer(&u.data[0])) }
func (u *Union) SetFloat64(val float64) { *(*float64)(unsafe.Pointer(&u.data[0])) = val }
func (u *Union) GetFloat64() float64    { return *(*float64)(unsafe.Pointer(&u.data[0])) }
` ``

## Риски использования `unsafe`

| Риск | Описание |
|------|----------|
| **Утечки памяти** | Неправильное освобождение памяти приводит к её утечке |
| **Висячие указатели** | Использование указателей на уже освобождённую память вызывает неопределённое поведение |
| **Нарушение типобезопасности** | Некорректное приведение типов может привести к непредсказуемым результатам и крашам |
