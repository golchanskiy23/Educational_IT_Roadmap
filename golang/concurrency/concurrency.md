# Concurrency Go

## Главное преимущество
Упрощение разработки многопоточных программ в современной многопроцессорной среде.

## Горутины
Goroutines, представляющие собой легковесные потоки выполнения, обеспечивают значительные преимущества по сравнению с традиционными потоками. Это способ выполнять несколько функций конкурентно внутри одной программы. Они позволяют создавать тысячи параллельных процессов без значительной нагрузки на системные ресурсы.

### Стек
По умолчанию каждая Goroutine начинается с небольшого размера стека, обычно несколько килобайт. Это существенно меньше, чем стековый размер стандартных потоков операционной системы, что позволяет создавать тысячи или даже миллионы Goroutines в одном приложении с минимальным потреблением памяти. Стек Goroutine может динамически расширяться и сжиматься в зависимости от потребностей. Это означает, что Goroutine использует только тот объем памяти, который ей действительно необходим, и может адаптироваться к изменяющимся требованиям памяти во время выполнения.

- Переменные, выделенные в Goroutine, которые должны сохраняться после завершения функции, автоматически перемещаются на кучу. Это управление памятью выполняется автоматически компилятором Go.

- Выделяемый стек - 2 кБ из runtime/stack.go
- Стек выделяется из кучи, при этом в отличии от потоков стек горутины сосуществует рядом с heap. Что-то вроде

[ stack ↓ ]
   ...
[ guard page ]
   ...
[ heap ↑ ]

- Итак изначально память выделяется из кучи, когда оперативка заканчивается, ОС использует файл подкачки, используя память на диске, тем самым замедляя выполнение программы, приводя машину к смерти.

- Стековый рост(стек может расти/уменьшаться):
Проверка размера стека
Если места недостаточно, вызывается morestack, который выделяет больший стек, перемещает в него старый, обновляет указатели и возвращает выполнение. Если используется меньше чем на 1/4, то копирование на меньший.
- При этом рантайм устанавливает и верхний предел — горутина не может превысить 1 ГБ на 64-битных системах и 250 МБ на 32-битных. Если предел достигнут, вызывается runtime.abort и программа падает с stack overflow.
```go
// в самом go есть структура, указывающая на начало и конец стека
// используется в определении стека горутины внутри struct g
type stack struct {
	lo uintptr
	hi uintptr
}
```

### Ограничения
- Несмотря на возможность создания большого числа горутин память компьютера не бесконечна и при дохождении до лимита RAM страницы памяти выгружаются на диск(подкачка) -> выполнение замедляется.
- Паузы в Go GC связаны с фазами stop-the-world, в которых происходит сканирование корневых ссылок на объекты в куче и, иногда, при завершении фазы маркировки(Mark termination). Хотя стек участвует в root scanning, основная стоимость GC обычно связана с обработкой кучи, а не самих стеков.

- Решения:
    Ограничение размера пулом воркеров
    Мониторинг(Prometheus)
    Уменньшение числа аллокаций

### Contiguous stack
Когда стек реально расширяется, рантайм копирует его содержимое в новый блок большего размера и исправляет все внутренние указатели, чтобы выполнение продолжилось без ошибок. Внешний код никогда не получает указатель внутрь стека горутины. Перед каждым вызовом функции Go проверяет, достаточно ли места на стеке. Если нет — вызывается runtime.morestack, который выделяет новую страницу, и только после этого функция выполняется.
Ранее применялся подход с сегментированными стеками, т.е. при необходимости выделялся новый стек, в итоге вместо одного получали стек функции, разбитый по нескольким стекам(linked list стеков).

### Утечка горутин
Если горутина блокируется и не завершается, но при этом продолжает занимать память.


## Каналы
- Предоставляют мощный механизм коммуникации между горутинами. Отправка данных в канал в Go осуществляется с помощью оператора <-. 
- Ключевая особенность работы с каналами в Go — возможность использования оператора select для мультиплексирования операций отправки и приема по нескольким каналам. Это позволяет Goroutine ожидать на нескольких каналах одновременно, продолжая выполнение, как только одна из операций становится возможной.

- Функция close(ch) используется для закрытия канала. После закрытия канала нельзя отправлять в него данные, но можно продолжать читать данные, которые были в нем до закрытия.

### Небуферизированные каналы
Когда Goroutine отправляет данные в небуферизированный канал, она блокируется до тех пор, пока другая Goroutine не прочитает эти данные. Аналогично, если Goroutine пытается прочитать данные из канала, она будет блокирована, пока другая Goroutine не отправит данные.

### Буферизированные каналы
Буферизированный канал имеет внутренний буфер, позволяющий отправлять данные в канал без блокировки до тех пор, пока буфер не заполнится. Отправка в полностью заполненный буферизированный канал блокирует Goroutine, пока в канале не появится свободное место.

## Синхронизация данных
### Мьютексы
Мьютекс используется для обеспечения взаимного исключения доступа к общим ресурсам, тем самым предотвращая одновременное изменение данных несколькими Goroutines.
```go
var mu sync.Mutex
mu.Lock()   // Блокировка мьютекса перед доступом к общим данным
// Критический раздел: здесь происходит работа с общими данными
mu.Unlock() // Разблокировка мьютекса после завершения работы
```

### RW-мьютекс
Read-Write Mutex (представленный как sync.RWMutex) позволяет множественный доступ для чтения, но исключительный доступ для записи. Множество Goroutines могут одновременно захватывать мьютекс в режиме чтения. Только одна Goroutine может захватывать мьютекс в режиме записи, и при этом должна быть гарантирована полная исключительность (никакие другие Goroutines не могут читать или писать).

```go
import "sync"

var rw sync.RWMutex
var sharedResource int

func readOperation() {
    rw.RLock() // Захват мьютекса для чтения
    _ = sharedResource // Чтение общего ресурса
    rw.RUnlock() // Освобождение мьютекса после чтения
}

func writeOperation() {
    rw.Lock() // Захват мьютекса для записи
    sharedResource++ // Модификация общего ресурса
    rw.Unlock() // Освобождение мьютекса после записи
}
```

### Каналы как синхронизация
Небуферизированные каналы
```go
ch := make(chan int)
   go func() {
       // выполнение задачи
       ch <- result
   }()
```

### Wait groups
sync.WaitGroup используется для ожидания завершения группы Goroutines. Полезно для сценариев, где нужно дождаться завершения всех запущенных Goroutines, прежде чем продолжить выполнение программы. Каждая Goroutine при запуске вызывает wg.Add(1), а по завершении — wg.Done(). Главная Goroutine использует wg.Wait() для ожидания завершения всех Goroutines.

```go
   var wg sync.WaitGroup
   wg.Add(1)
   go func() {
       defer wg.Done()
       // выполняемая задача
   }()
   wg.Wait()
```

### Атомарные операции
Атомарные операции позволяют выполнять операции над общими данными без необходимости использования мьютексов.
```go
    var counter int32
    atomic.AddInt32(&counter, 1)
```
